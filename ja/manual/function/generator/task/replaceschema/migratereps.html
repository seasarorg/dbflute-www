概要

既に開発中、運用中のシステムなどで、テストデータをテストDBの中でしか管理しておらず、 かつ、そのテストデータがテストをやる上で重要というような場合が多々あります。 そのような状態から ReplaceSchema に移行するには、テストデータをテストDBから何かしらの方法で抽出して、ReplaceSchema でデータ登録できる形にする必要があります。

そこにはどうしても手作業が入らざるを得ないことに変わりはありませんが、DBFluteではそれを支援する機能があります。 テストDBからテストデータを抽出し、FK制約の順序をある程度考慮した上でエクセルデータとして playsql 配下に出力することができます。あとは、その抽出されたエクセルデータの名前やシート構成などを人が手動で整理して、ReplaceSchema 管理へと移行します。@since 0.9.8.3

データの抽出だけを対象としてDDLなどはスコープ外なので、それは各自準備する必要があります。

MigrateRepsタスク

データを移行するための機能は、DBFluteタスクとして用意されています。

事前にJDBCタスク

まずは、JDBCタスクが実行されていることが前提条件となります。これは、内部的な処理の中でFK制約情報などのメタ情報を利用するためです。

タスク実行スクリプト作成

通常、DBFluteタスクの実行スクリプト(generate.bat|sh など)は既に用意されています。MigrateRepsタスクは、一発だけの実行、かつ、限定的な状況で利用されることを想定しているので、 実行スクリプトはデフォルトでは存在しません。

他のタスクの実行スクリプトをコピーして書き換えて作成します。具体的には、内部的なスクリプトを呼び出している部分を _df-migrate-reps.bat|sh に書き換えます。 

ex) 他のタスクの実行スクリプトをコピーしてMigrateReps用に修正 (シェル: Mac OS X) @migrate-reps.sh
#!/bin/bash

cd `dirname $0`
. _project.sh

echo "/nnnnnnnnnnnnnnnnnnnnnnnnnnnnnn"
echo "Execute the MigrateReps task."
echo "nnnnnnnnnn/"
sh $DBFLUTE_HOME/etc/cmd/_df-migrate-reps.sh $MY_PROPERTIES_PATH
taskReturnCode=$?

if [ $taskReturnCode -ne 0 ];then
  exit $taskReturnCode;
fi

MigrateReps実行

自作の実行スクリプト経由で、MigrateRepsタスクを実行します。ReplaceSchema のときと同様、本当に実行するのかどうかを y/n 形式で聞かれますので、y と押してください。

タスクログ(コンソール or dbflute.log)に、抽出されたのテーブルとレコード件数が表示され、 エクセルデータが作成されます。(Eclipse上のリフレッシュは手動で)

出力先ディレクトリ
DBFluteクライアント/playsql/data/[現在のdataLoadingType]/firstxls
dataLoadingTypeは、デフォルトでは ut
ファイル名
migration-data-section[セクション番号].xls
セクション番号は 01 始まり

firstxls ディレクトリに出力されます。これは、通常の xls ディレクトリとは違い、TSVやCSVのデータ登録よりも先に実行されるエクセルデータの配置領域です。

データ登録(エクセル) - エクセルファイルの配置

セクションは、FK制約の階層構造を参考に、他のテーブルへの依存の少ないテーブルから上位のセクションに割り振られます。 よって、厳密ではありませんが、FK制約の階層構造のレイヤ数の分、セクション(エクセルファイル)が作成されます。 例えば、ExampleDB では、2 階層となり二つのセクションができあがります。実業務の複雑なテーブル構造では、セクションが 7, 8 ともっと増えることが想定されます。(10 を超えることもあるでしょう)

手動による調整

この時点で、シンプルなテーブル構造、シンプルなデータであれば、このまま ReplaceSchema でデータ登録することも可能です。firstxls なので、TSVやCSV、通常のxlsよりも先に登録されます。 ただ、実業務では多くの場合そのようにはならないでしょうし、セクションの割り振りなどは業務上のカテゴリを無視してものとなっているため、 実際には手動で調整する必要があるでしょう。

  • 業務カテゴリごとにエクセルファイルを構成し直す (ファイル名変えたり、シートを移動したり)
  • FK制約の階層をどうしても自動で解決できなかったものの順序を調整
  • (必要に応じて) firstxls から xls に移動 (メインは xls なので)
  • この機能でサポートされていないデータ型のデータを調整 (画像などのバイナリ型など)

これらの調整ができて、DDLなどの他の要素の準備も整ったら、ReplaceSchemaへの移行完了です。

細かい仕様

細かい仕様を把握して、どのような調整を行えば良いのか判断していきましょう。

サポートされるレコード数は 65000 まで
DBFluteでサポートしているエクセルの行数の限界が 655xx なので、MigrateRepsタスクで出力されるレコード数の最大は 65000 と定義されています。それ以上のレコードは切り捨てられます。
バイナリ型のデータは全て null で出力
バイナリ型のデータはサポートされず、全て null として出力されます。
テストデータとして必要な場合は、手動で準備します。
データ登録 - バイナリデータの登録
厳密な精度は保証されない
例えば、日付型は以下のような形式で出力されます。
  • DATE : yyyy/MM/dd (Oracleは HH:mm:ss も追加)
  • TIMESTAMP : yyyy/MM/dd HH:mm:ss.SSS
  • TIME : HH:mm:ss
もし、TIMESTAMP型がこの書式を超える細かい精度を持っていると、その精度は失われます。
その他の型は、単なるJDBCドライバの時点で文字列に変換したものを利用します。 JDBCドライバや文字列に直した時の特徴的な仕様によって、精度が失われる可能性はあります。
これら精度を厳密に移行したい場合は、手動で調整する必要があります。 また、あくまでテストデータの移行というのを大前提としているので、実業務の本番データの移行などでは利用できません。
その他、メジャーでない型はサポートされない
例えば、PostgreSQL の OID 型など、取扱いの難しいものなどはサポートされません。 明示的にサポートしていないわけではなく、そういった型がある場合、単にJDBCドライバの時点で文字列に変換したものが出力されます。 よって、JDBCドライバがその文字列を再解析できる場合は、ReplaceSchema でその文字列のままで登録できるかもしれません。
共通カラムも出力される
ReplaceSchema には共通カラムを固定的に登録する機能があり、エクセルデータ上には共通カラムがなくてもデータ登録はできますが、 この機能では共通カラムのデータも一緒に出力されます。既にテストDBとして存在していたということは、 共通カラムもそのデータのスナップショットを表す大事な一つの要素と考えられるからです。
