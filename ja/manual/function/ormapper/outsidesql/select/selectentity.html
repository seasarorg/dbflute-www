概要

外だしSQLで null を戻す可能性のある一件検索 をします。検索結果が存在する場合としない場合で業務的な処理を分岐させる必要があるような場合に利用します。

(検索結果が)存在しなかったら例外になってもいい(それは何かの間違い)、という場合は、このメソッドではなく、selectEntityWithDeletedCheck() を利用して下さい。

selectEntityWithDeletedCheck()

会話上では、(外だしSQLの)せれくとえんてぃてぃ と表現します。

実装方法

メソッドの呼び出し

outsideSql() の後、entityHandling() を呼び出し、その後 selectEntity() を呼び出します。 

ex) 外だしSQLの一件検索の実装手順 (Eclipseでコード補完) @Java
memberBhv.o // .o と打って enter
--
memberBhv.outsideSql().en // .en と打って enter
--
memberBhv.outsideSql().entityHandling().selE // .selE と打って enter
--
memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType);

SQLのパス・パラメータ

第一引数のSQLのパス、第二引数の ParameterBean に関しては、基本仕様と全く同じです。

外だしSQLの使い方 - 基本仕様

検索結果の受け取り

検索結果は戻り値で、第三引数で指定した Entity のインスタンスとして受け取ります。 

ex) 外だしSQLの一件検索の実装 @Java
String path = MemberBhv.PATH_selectSimpleMember;
SimpleMemberPmb pmb = new SimpleMemberPmb();
pmb.setMemberId(3);
Class<?> entityType = SimpleMember.class;
SimpleMember member = 
    memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType);

検索結果が存在しない場合

検索結果が存在しない場合の戻り値は null です。戻り値を null かどうかで処理を分岐させます。 

ex) 外だしSQLの一件検索で戻り値の有無で分岐 @Java
...
SimpleMember member = 
    memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType);
if (member != null) {
    // if exists
} else {
    // if NOT exists
}

検索結果が複数ある場合

指定された条件で検索結果が複数ある場合は、条件もしくはSQLの不備と判断され、EntityDuplicateException が発生します。 

ex) 外だしSQLの一件検索で条件なしで検索して結果が複数件取れちゃった @Java
String path = MemberBhv.PATH_selectSimpleMember;
SimpleMemberPmb pmb = new SimpleMemberPmb();
Class<?> entityType = SimpleMember.class;
try {
    memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType)
} catch (EntityDuplicateException e) {
    ...
}

スカラ値で受け取る

SQL上の select 句のカラムを一つにして、検索結果を String や Integer などのスカラ値で受け取ることも可能です。(これらも含めての広い意味で (select)Entity として扱っています)

検索件数の取得 "select count(*)" や最大値の取得 "select max(xxx)" などにおいて有効です。 

ex) 文字列のスカラ値を検索 (nullの可能性あり) @Java
...
Class<?> entityType = String.class;
String value = 
    memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType)
// select MEMBER_NAME from ... where MEMBER_ID = ...

ex) 数値のスカラ値を検索 (nullの可能性あり) @Java
...
Class<?> entityType = Integer.class;
Integer value = 
    memberBhv.outsideSql().entityHandling().selectEntity(path, pmb, entityType)
// select max(MEMBER_ID) from ...

スカラ値での検索においても、検索結果がない場合の扱いについては全く同じです。

なぜ、一歩手順が多いの？

selectList()とは並ばず、entityHandling()って一歩進めないといけない外だしSQLの一件検索。 なぜかと言うと、ズバリ、(作者の経験上)実務的に外だしSQLのほとんどがリスト検索になりやすいと判断したからです。DBFluteには強力な ConditionBean があるため、特に一件検索はほとんどCBで実装できることが多いためです。その分、外だしSQLのリスト検索は outsideSql() の後に、唯一の "s" で始まるメソッドしてコード補完しやすくなっています。
