概要

"JSON APIのJSONデザイン" で、以下のものを採用したときのパターンを紹介します。 (身の回りでよく使われるでありながら、よく考えてデザインしてクライアントと共有しないと、"へんてこりん" になりがちだし、実装も少し工夫が必要なので)

  • B. エラー表現: Failure統一パターン
  • E. メッセージ管理: クライアントメッセージ管理

これを ふぇいくらパターン とここでは呼びます。

JSON APIのJSONデザイン

ちなみに、このページの Example では、キーのケースは "スネークケース" にしています。

JSONのイメージ

e.g. "ふぇいくらパターン" で、バリデーションエラーのとき @Json
{
    cause : (VALIDATION_ERROR or BUSINESS_ERROR
            or CLIENT_ERROR or SERVER_ERROR)
    , errors : [
         , field : (受け取ったJSON項目名、階層表示あり e.g. product_name)
         , code : (エラーチェックの種別を示すコード e.g. REQUIRED, MAX など)
         , values : (エラーのきっかけとなった定義値のリスト
                    e.g. 数値チェックの最大値、長さチェックの最大長など)
    ], [
        ...
    ]
}

バリデーションエラーのJSON

  • field は、ユニークとは限らない (一つの項目で二つエラーになり得るため)
  • field は、階層構造を表現 (受け取ったJSONの階層構造をドット区切りで表現)
  • code は、チェック処理の種別ごとで、基本的にValidatorアノテーションごと
  • values は、チェック処理で使った定義値 (メッセージに埋め込まれることを想定)
e.g. "ふぇいくらパターン" で、バリデーションエラーのとき @Json
{
    cause : "VALIDATION_ERROR"
    , errors : [
         , field : "product_name"
         , code : "REQUIRED"
         , values : []
    ], [
         , field : "product_count"
         , code : "MAX"
         , values : ["100"] // 数値の最大値
    ]
    ], [
         , field : "member.email"
         , code : "LENGTH"
         , values : ["50"] // 文字列の最大長
    ]
    ], [
         , field : "member.email"
         , code : "EMAIL"
         , values : []
    ]
}

ビジネスエラーのJSON (業務例外)

  • field は、_global 固定 (特定の項目がないため) (LastaFluteのデフォルト)
  • code は、業務例外の種別を特定
  • values は、基本的に使うことはないが、必要なこともあるかもしれないので
e.g. "ふぁいくらパターン" で、業務例外のとき @Json
{
    cause : "BUSINESS_ERROR"
    , errors : [
         , field : "_global"
         , code : "ALREADY_DELETED"
         , values : []
    ]
}

cause が直接 ALREADY_DELETED になるやり方もあり得ますが、フラット構造になってしまってクライアントサイドが分岐をしづらいかもしれませんし、values をどうやって渡すかを工夫しないといけないでしょう。

field の _global は、実質クライアントは見ないかもしれません。BUSINESS_ERROR だったら、特定の項目に対するメッセージじゃないって割り切った判断をするかもしれませんので。 万が一、項目特化の業務例外があったときのためのものです。

ちなみに業務例外の粒度ですが、これは "ふぇいくらパターン" に限らず、例外ハンドリングの "一件検索でデータ無かったとき" のページを参考に。

開拓の例外ハンドリング - 一件検索でデータ無かったとき

クライアントエラーのJSON (クライアント例外)

基本的には、クライアント例外はクライアントにとっての不具合なので、発生したらサーバーサイドのログを見て状況を把握してクライアントプログラムを修正します。 (HTTPステータスが 400, 403, 404 の違いで若干状況は把握できますが、あまりデバッグとしてはそこまで重要ではないでしょう)

クライアント例外ハンドリング

ゆえに、クライアント例外は、多くのケースでメッセージを必要としないので、errors は空っぽです。 

e.g. クライアント例外のとき @Json
{
    cause : "CLIENT_ERROR"
    , errors : []
}

サーバーエラーのJSON (システム例外)

e.g. サーバー例外のとき @Json
{
    cause : "SERVER_ERROR"
    , errors : []
}

LastaFluteでの実装

Failure統一パターンは、Exampleと同じ

Failure統一パターンというところは、LastaFlute の Example と同じなので、harbor や maihama プロジェクトの ApiFailureHook を参考に実装すればOKです。

HarborApiFailureHook.java | Github
忘れられないApiFailureHook

ただ、メッセージは、サーバーメッセージ管理方式なので、その辺を微調整する必要があります。

errors の 実装をどうしよう？

errors.code や errors.values をどうやって定義して取得するかを少し考える必要があります。

というのは、Hibernate Validator からは、REQUIRED や MAX などの種別情報は取得できず、[app]_message.properties のメッセージしか取得できないからです。(constraints.Required.message ではなく、"is required" しか取得できない)

そこで、[app]_message.properties のメッセージに、code と values のために値を入れておきます。 もうサーバーサイドでメッセージは管理しないので、自然言語部分は削除します。 

e.g. "ふぁいくらパターン" のために [app]_message.properties のバリデーションメッセージを修正 @Properties
# (key of message) = (errors.code, errors.values)
...
constraints.Max.message = MAX, {value}
constraints.Min.message = MIN, {value}
...
constraints.Length.message = LENGTH, {min}, {max}
...
constraints.Required.message = REQUIRED
constraints.TypeAny.message = TYPE_ANY, {propertyType}
constraints.TypeInteger.message = TYPE_NUMBER // これ新しく追加
constraints.TypeLong.message = TYPE_NUMBER // これ新しく追加
constraints.TypeLocalDate.message = TYPE_DATE // これ新しく追加
...

データ型変換エラーで、Integer や LocalDate など Java に依存した値を戻しても良くないので、 必要に応じて、TypeInteger, TypeLocalDate などのメッセージを追加すると良いでしょう。 (TypeAnyは、データ型変換エラーのデフォルトのメッセージという扱い)

バリデーションエラーの種別 (Required, Max など) と、errros.code は必ずしも同じ値でしなくても良いですが、基本的には合わせておいたほうが間違いは少ないでしょう。 ただ、Integer か Long や LocalDate をクライアントに意識させる必要もないので、TYPE_NUMBER や TYPE_DATE という風に少し翻訳しています。

業務例外で追加するメッセージも、同じような構成で定義します。errors.code は業務例外の種別ごとに考えて定義しましょう。つまり、サーバーサイドの中では、"メッセージ" というのは実質的に "code と values" を指します。 (ちなみに、[app]_label.properties は基本的に使うことがないです) 

e.g. "ふぁいくらパターン" のために [app]_message.properties の業務例外メッセージを修正 @Properties
...
# ----------------------------------------------------------
#                                      Application Exception
#                                      ---------------------
# /- - - - - - - - - - - - - - - - - - - - - - - - - - - - -
# six framework-embedded messages (don't change key names)
# - - - - - - - - - -/
errors.login.failure=LOGIN_FAILURE
errors.app.illegal.transition=ILLEGAL_TRANSITION
errors.app.db.already.deleted=ALREADY_DELETED
errors.app.db.already.updated=ALREADY_UPDATED
errors.app.db.already.exists=ALREADY_EXISTS
errors.app.double.submit.request=DOUBLE_SUBMIT


# _/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/
# you can define your messages here:
# e.g.
#  errors.xxx = ...
#  info.xxx = ...
# _/_/_/_/_/_/_/_/_/_/

errors....=...
...

ApiFailureHookにて、この "メッセージ" (errors.code, errors.values) をパースして、JSON Result に設定するロジックを実装しましょう。

クライアントエラーの実装は？ (クライアント例外)

causeだけを CLIENT_ERROR にして、errorsの実装は実質的に業務例外と同じで良いでしょう。メッセージがあればが設定されるし、なければ空っぽになるだけなので。

ただ、アプリケーションサーバーレベルの 404 なども JSON でしっかり戻すのであれば、web.xml にて定義されている 400.html, 404.html などを 400.json, 404.json に変更して、アプリが戻すクライアント例外のJSONと一致する固定のJSONを定義しておくと良いでしょう。

サーバーエラーの実装は？ (システム例外)

ApiFailureHookでは何もせず(OptionalThing.empty()を戻す)、web.xml にて定義されている 500.html を 500.json に変更して、アプリが戻すJSONと一致する固定のJSONを定義しておくと良いでしょう。

ただ、そもそもクライアントは、"HTTPステータスが 500 だったら中身も見ずにシステムエラーの処理をするだけ" という振舞いが想定されるので、あまり意味はないかもしれません。
