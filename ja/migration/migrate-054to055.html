【S2Daoの最新】

対応するS2DaoのDefaultは、1.0.46となります。
それ以前のS2Daoを利用する場合は、

  torque.s2daoVersion = 1.0.45

というように指定して下さい。


【[DBFLUTE-144]】

お手数ですが特に問題がないようであれば、以下のPropertyを定義して[true]に設定して頂けませんでしょうか？

*** 利用者視点の動作は全く変わりません。***
内部の動きが少し変わり、BehaviorのdelegateXxx()メソッドがprotectedに変わります。

# /---------------------------------------------------------------------------
# [Common Column Intercepting On Behavior Filter] (Default 'false' *But at the future it will be true!)
#   [true]
#     Common column intercepting is executed on behavior filter.
#     And the modifier of behavior's delegateXxx() method is protected!
#
#   [false]
#     Non. (Common column intercepting on interceptor and delegateXxx() is public)
#
# @JavaOnly
torque.isCommonColumnInterceptingOnBehaviorFilter = true
# ----------------/

今まで、共通列(CommonColumn)の自動設定をInterceptorにて行っており、
BehaviorのdelegateXxx()メソッドに関連付けて実行していました。
ですが、その方法は以下の問題が発生します。

  A. BehaviorのdelegateXxx()が内部メソッドなのにpublicなのが紛らわしい。
  B. Dicon上のInterceptorのAspect関連付けがDicon初期化時に遅い。

「A」は、特に初めて利用される人にとって紛らわしいことが多いようです。
知らずにdelegateXxx()を使っていると、Behaviorの提供するメソッドの
細かいメリットが享受できないため、あまりよくありません。
このメソッド群は、protectedであるべきですが、
Interceptor方式だとpublicにせざるを得ないのです。
(Interfaceを用意する方法もありますが、BehaviorにはInterfaceを付けたくない気持ちがあります)

「B」は、結構明らかに違います。
特にDB周りの単体テストを行う場合に、そのスピードが全然違います。
単体テストの初期化が遅いと単体テストもやる気が失せやすいものです。

そこで、自動生成の利を活かして、BehaviorのdelegateXxx()内のFilterにて
共通列(CommonColumn)の自動設定を行うことにより、上記問題が解消します。
「もともと自動生成なんだからInterceptorで実現する必要はない」
という声も頂き、今回このように判断しました。

「DBFlute-0.6.0」において、さらにこのPropertyのDefaultを「true」に
させて頂きたいと考えています。「false」に基本的に下位互換専用ということで。
