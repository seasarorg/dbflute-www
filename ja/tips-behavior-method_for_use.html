BehaviorのMethodの用途別概要を記述します。 



// ======================================================================================================
//                                                                            Behavior - 用途別Method概要
//                                                                                  =====================

# -------------------------------------------------
#                                        とりあえず
#                                        ----------

とりあえずは、bhv.selectS...まで書けば、おおよそ勘で利用できるSimpleなMethodが補完されるはずです。

  selectSimpleCount();
  selectSimpleEntity();
  selectSimpleList();
  ...


また、bhv.delegate...まで書けば、Daoに定義されているMethodへのDelegate-Methodが補完されるはずです。

  delegateInsert()
  delegateUpdate()
  ...



# -------------------------------------------------
#                                      Select Count
#                                      ------------

ConditionBeanによるCount検索：

    → selectSimpleCount()

      もし、引数のConditionBeanに fetchFirst()やFetchPage()などが呼び出されていて
      ConditionBeanのFetchScope設定が有効な場合、このMethodはその設定を無視して処理します。
      例えば、342件取得できるはずのConditionBeanの検索条件にfetchFirst(20)/fetchPage(5)と設定されていた場合
      このConditionBeanを selectSimpleCount(cb); で呼び出すと、342件が取得されます。

        ※DBFluteの独自用語として、このことを IgnoreFetchScope と言います。



# -------------------------------------------------
#                                     Select Entity
#                                     -------------


ConditionBeanによるEntity検索：

    → selectSimpleEntity()

      該当の条件にHITするRecordが無い場合はNullが戻ります。
      該当の条件にHITするRecordが複数件の場合は例外が発生します。


ConditionBeanによる「nullを期待しない」Entity検索：

    → selectEntityWithDeletedCheck()

      該当の条件にHITするRecordが無い場合は例外が発生します。
      該当の条件にHITするRecordが複数件の場合は例外が発生します。


PrimaryKeyによる「nullを期待しない」Entity検索：

    → selectByPKValueWithDeletedCheck()

      selectEntityWithDeletedCheck()の引数がPrimaryKeyになったものです。
      オーバーロードは利用せずに、明示的にMethod名を変えています。


PrimaryKeyによる「更新Lock付きでnullを期待しない」Entity検索：

    → selectForUpdateByPKValueWithDeletedCheck()

      selectByPKValueWithDeletedCheck()に更新Lockを付与したものです。


_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/

delegateSelectEntity()を直接呼び出すことはお奨めしません。
delegateSelectEntity()はS2DaoのMethodをそのまま呼び出しています。
S2Dao仕様では、該当条件に複数件HITする場合に先頭の一件目を戻すようになっています。
これは便利な時もありますが、思わぬBUGを生み出すこともあります。

なので、安全のためにBehaviorが提供するEntity検索Methodは、全て複数件HIT時に例外を発生させます。
もし、「明示的に複数件HITする中での先頭の一件が欲しい」という要件がある場合は、
以下のようにConditionBeanを設定して下さい。

    /- - - - - - - - - - - - - - - - - - - - - - - - - - 
    LdBookCB cb = new LdBookCB();
    cb.query().setXxx...(xxx);
    cb.query().addOrderBy_Xxx_Asc();
    fetchFirst(1); // ☆
    LdBook entity = bhv.selectSimpleEntity(cb);
    - - - - - -/

すると、複数件HITする条件の中のSortされた上での一件目が取得できます。

# -------------------------------------------------
#                                       Select List
#                                       -----------


ConditionBeanによるList検索：

    → selectSimpleList()

      戻り値：java.util.List


ConditionBeanによる「ListResultBeanを戻り値とした」List検索：

    → selectList()

      戻り値：ListResultBean


ConditionBeanによる「PagingResultBeanを戻り値とした」Paging検索：

    → selectPage()

      戻り値：PagingResultBean



# -------------------------------------------------
#                                     Entity Insert
#                                     -------------

通常の登録：

    → delegateInsert()

      該当のRecordが既に存在する場合は一意制約違反のSQLExceptionが発生します。


なければ登録、あれば更新：

    → insertOrUpdateAfterSelect()

      主に外部SystemからのDataを夜間に取り込むBatchなどで利用される。


なければ登録、あれば更新｛更新Lock付き｝：

    → insertOrUpdateAfterSelectForUpdate()

      主に外部SystemからのDataを夜間に取り込むBatchなどで利用される。


# -------------------------------------------------
#                                     Entity Update
#                                     -------------

通常の更新：

    → delegateUpdate()

      S2Dao-1.0.39以前：
        Entityに定義されている全てのColumnがUpdate文のSet句に指定されます。
        引数のEntityに値がSetされていないColumnは、全てnullで更新されます。

        引数のEntityは、newして値をSetしたものではなく、一度SelectしたEntityを
        利用するのが良いかと思われます。

      S2Dao-1.0.40以後：
        EntityのSetterが呼び出されたColumnだけがUpdate文のSet句に指定されます。

        引数のEntityは、newして値をSetしたもので構いません。
        但し、VersionNoやTimestampをEntityに設定している場合で、引数のEntityのそれらの値が null の場合は、
        0件更新となり更新に失敗します(S2Daoの仕様)。


一旦Selectしてから更新：

    → updateAfterSelect()

      S2Dao-1.0.39以前：
        Entityに定義されている全てのColumnがUpdate文のSet句に指定されます。
        引数のEntityに値がSetされていないColumnは、全てDB上の値そのままで更新されます。

        引数のEntityは、newして値をSetしたもので構いません。
        但し、VersionNoやTimestampをEntityに設定している場合で、引数のEntityのそれらの値が null の場合は、
        DB上の値そのままを排他制御に利用するため、実質的な排他制御が行われずに更新されます。
        (逆に排他制御をしたくない場合はこのMethodを使うと良いかと思います)

      S2Dao-1.0.40以後：
        EntityのSetterが呼び出されたColumnだけがUpdate文のSet句に指定されます。

        引数のEntityは、...(S2Dao-1.0.39以前の記述と同様)


更新Lock付きで一旦Selectしてから更新：

    → updateAfterSelectForUpdate()

      ForUpdateが付いただけで、それ以外の動きはupdateAfterSelect()と同様です。



_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/_/

検索と登録に関しては特に迷うところはないかと思いますが、
更新は複雑なためどのMethodを使えば良いのか迷います。

Projectによって様々ではありますが、以下の点を割り切りできるならば、
基本的に updateAfterSelect() のみを利用すれば良いかと思います。
(大人数で開発する時などProgramerを迷わせないために「これを使う」と決めてしまう)


  A. VersionNo or Timestampを引数のEntityにSetし忘れた場合は、
     排他制御がされない。(排他制御を強制しない)

  B. Update前にSelectするCost。

  C. Update前にSelectは更新Lockなし。


「A」に関して割り切れない場合は、やはりdelegateUpdate()を利用して下さい。
updateAfterSelect()は排他制御が業務要件的に不要である場合にだけ利用して下さい。

「B」に関して割り切れない場合は、やはりdelegateUpdate()を利用して下さい。
updateAfterSelect()は排他制御が業務要件的に不要である場合にだけ利用して下さい。

「C」に関して割り切れない場合は、updateAfterSelectForUpdate()を利用して下さい。



#
# 別の視点からの判断基準の一つ：
#
# もし、排他制御の結果として、「そのRecordが他の人に削除されたのか更新されたのか」
# を区別して扱いたい場合は、updateAfterSelect() or updateAfterSelectForUpdate()
# を利用しなければなりません。(delegateUpdate()ではその区別は付けられない)
#
