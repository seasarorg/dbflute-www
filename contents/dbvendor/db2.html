DBFluteにおけるDB2の扱い

概要

DBFluteを利用する上でのDB2の扱いを述べます。

※バージョンはDB2-8以上を前提として記載しています。

JDBCタスクで利用するJDBCドライバ

JDBCドライバは「db2jcc.jar(推奨)」ならびに「db2java.zip」を利用しています。

DBFlute-0.6.1までをご利用の方は「db2jcc.jar」は同梱されていませんので、「mydbflute/dbflute-x.x.x/lib/forExecute」配下に手動で配置して頂くか、「db2java.zip」の方をご利用して頂く(driver名を変える)ことになります。

「db2jcc.jar」を利用する際に、DB2のバージョンによっては「db2jcc_license_cu.jar」が必要になることがあります。もし、そのjarが必要な場合は「mydbflute/dbflute-x.x.x/lib/forExecute」配下に手動で配置して下さい。(DB2 9.5/Linux(2008年1月頃の最新)では不要であるという報告をユーザの方から頂いております)

JDBCタスクの接続設定(./dfprop/databaseInfoMap.dfprop)

JDBCタスクの接続設定は以下の通りです。スキーマ名は必ず大文字で指定して下さい。 

ex) 接続{port=50000, db2jcc.jar/db2jcc_license_cu.jar利用} @databaseInfoMap.dfprop
map:{
    ; driver   = com.ibm.db2.jcc.DB2Driver
    ; url      = jdbc:db2://localhost:50000/[dbname]
    ; schema   = [dbuser-as-UpperCase]
    ; user     = [dbuser]
    ; password = [dbpassword]
}
	
ex) 接続{port=50000, db2java.zip利用} @databaseInfoMap.dfprop
map:{
    ; driver   = COM.ibm.db2.jdbc.app.DB2Driver
    ; url      = ...
	; ...
}

主キーで連番の利用

DB2は連番生成の仕組みとしてSequenceとIdentityを利用することができます。

DBFluteでのSequence設定

詳しくはDBベンダごとの扱いの「主キーの連番でSequenceを使うDBの場合」をご覧下さい。以降はDBベンダ依存の補足です。

補足：DB2でSequenceを作成

Create Sequence文では以下のように指定します。 

ex) DB2でSequenceを作成 @DDL
CREATE SEQUENCE SEQ_MEMBER AS INT START WITH 1 INCREMENT BY 1;

補足：DB2でSequenceを取得

Create Sequence文では以下のように指定します。 

ex) DB2でSequenceを取得 @SQL
values nextval for SEQ_MEMBER;

DBFluteでのIdentity設定

詳しくはDBベンダごとの扱いの「主キーの連番でIdentityを使うDBの場合」をご覧下さい。以降はDBベンダ依存の補足です。

補足：DB2でIdentity設定

Create文では以下のように指定します。 

ex) DB2でIdentity設定 {MEMBERのMEMBER_IDにIdentity} @DDL
CREATE TABLE MEMBER
(
    MEMBER_ID int generated by default as identity,
    MEMBER_NAME VARCHAR(200),
    ...
)

補足：DB2でInsert文の実行後に発行された連番を取得

Insert文の実行後に発行された連番を取得するSQLは以下のようになります。DBFluteを利用している場合は内部的に発行されるので意識する必要はありません。 

ex) DB2でInsert文の実行後に発行された連番を取得 @SQL
values IDENTITY_VAL_LOCAL();

ページングの実現方法

ConditionBeanではlimit処理としてfetch first n rows only構文を利用しています。offset処理はResultSetの読み飛ばしを利用します。将来的にrow_number関数を利用しての実現を検討しています(2008年1月頃)。 

ex) ConditionBeanでページング @SQL
    ... fetch first 100 rows only

OutsideSqlのManualPaging(手動ページング)を利用する場合は、row_number関数を使って以下のように実装します。 

ex) OutsideSqlのManualPaging(手動ページング)でページング @SQL
select *
  from (
  
    select ..., row_number() over(order by ...) as rn from ... where ...
	
       )
 where rn > /*$pmb.pageStartIndex*/80
   and rn <= /*$pmb.pageEndIndex*/100
 order by rn asc;

DB2でBLOG/CLOBを使う場合の注意点 @Java

DB2のJDBCドライバでは、ResultSetTypeがScrollableの場合にBLOBを取得できない(例外発生)という報告があります。DBFluteのデフォルトはScrollableなので、そのままでBLOB/CLOBの含まれているテーブルは検索できないことになりますが、その解決方法が２つあります(DBFlute-0.6.2より)。

１つ目は、ConditionBeanやOutsideSqlのconfigure()メソッドでStatementConfigを指定する方法です。 

ex) ConditionBeanでResultSetTypeをForwardOnlyに設定 @Java
MemberCB cb = new MemberCB();
cb.configure(new StatementConfig().typeForwardOnly());
... = memberBhv.selectList(cb);// ResultSetTypeはForwardOnlyで実行される
	
ex) OutsideSqlでResultSetTypeをForwardOnlyに設定 @Java
StatementConfig config = new StatementConfig().typeForwardOnly();
... = memberBhv.outsideSql().configure(config).selectList(...);// ResultSetTypeはForwardOnlyで実行される

２つ目は、DBFluteの全体ConfigとしてStatementConfigを指定する方法です。すると、DBFlute全体の検索のデフォルトが全て指定された通りになります。(ここいうデフォルトはConditionBeanやOutsideSqlのconfigure()メソッドが優先されることを意味します) 

ex) OutsideSqlResultSetTypeをForwardOnlyに設定 @Java
// ロックを外す
DBFluteConfig.getInstance().unlock();

// StatementConfig
DBFluteConfig.getInstance().setDefaultStatementConfig(new StatementConfig().typeForwardOnly());

// ...Containerの初期化 ※DBFluteConfigの設定はC必ずontainer初期化前にすること

局所的であれば、前者が推奨されます。しかし、アプリケーション全体で多くあるのであれば後者を利用して下さい。

後者のデメリットは、先述したページングのoffset処理の「ResultSetの読み飛ばし」が「ResultSetの空回し」になってしまうという点です。しかし、offset処理なので「1000ページ目(20000件付近)を検索」ということでない限り、ほとんどパフォーマンス劣化は気にするほどのものではないはずです。

どちらも使い方の詳しい実装例が、DBFluteBasicExampleにありますのでそちらぜひご覧になられて下さい。
