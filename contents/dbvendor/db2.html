DBFluteにおけるDB2の扱い

概要

DBFluteを利用する上でのDB2の扱いを述べます。

※バージョンはDB2-8以上を前提として記載しています。

JDBCタスクで利用するJDBCドライバ

JDBCドライバは「db2jcc.jar」を利用します。

JDBCドライバは同梱されていませんので、JDBCドライバのオフィシャルサイトからダウンロードして、 DBFluteクライアントの「extlib」配下に配置して下さい。(extlibディレクトリは手動で作成)

JDBCタスクの接続設定(./dfprop/databaseInfoMap.dfprop)

JDBCタスクの接続設定は以下の通りです。スキーマ名は必ず大文字で指定して下さい。 

ex) 接続{port=50000, db2jcc.jar利用} @databaseInfoMap.dfprop
map:{
    ; driver   = com.ibm.db2.jcc.DB2Driver
    ; url      = jdbc:db2://localhost:50000/[dbname]
    ; schema   = [dbuser-as-UpperCase]
    ; user     = [dbuser]
    ; password = [dbpassword]
}
	※[xxx]部分にプロジェクト独自の値を設定。[]で囲う必要はない。

主キーで連番の利用

DB2は連番生成の仕組みとしてSequenceとIdentityを利用することができます。

DBFluteでのSequence設定
詳しくはDBFluteのSequenceをご覧下さい。
DBFluteでのIdentity設定
後述の「DB2でのIdentityについて」をご覧下さい。

DB2でのIdentityについて

DB2のJDBCドライバにちょっとしたクセがあります。

Identityが「always」の場合は、特に問題ありません。DBFluteのIdentityをご覧下さい。

Identityが「by default」の場合は、DB2のJDBCドライバがオートインクメント情報を提供してくれないため(バグかどうか不明)、DBFluteに手動でIdentity情報を与える必要があります。 (※DBFlute-0.9.0/DBFlute-0.8.8.4/DBFlute.NET-0.8.8.4より「by default」も自動で認識できるようになりました) 

ex) 手動Identity設定 @identityDefinitionMap.dfprop
map:{
    MEMBER = MEMBER_ID
    ; PURCHASE = PURCHASE_ID
    ; ...
}

「always」の方が設定は楽になりますが、PKの値を明示的に指定したInsertができないためテストデータの管理等が煩雑になります。なので、「by default」がおすすめです。

補足：DB2でSequenceを作成

Create Sequence文では以下のように指定します。 

ex) DB2でSequenceを作成 @DDL
CREATE SEQUENCE SEQ_MEMBER AS INT START WITH 1 INCREMENT BY 1;

補足：DB2でSequenceを取得

Sequenceの値を取得するときは以下のようにSQLを記述します。 

ex) DB2でSequenceを取得 @SQL
values nextval for SEQ_MEMBER;

補足：DB2でIdentity設定

Create文では以下のように指定します。 

ex) DB2でIdentity設定 {MEMBERのMEMBER_IDにIdentity(by default)} @DDL
CREATE TABLE MEMBER
(
    MEMBER_ID int generated by default as identity NOT NULL PRIMARY KEY,
    MEMBER_NAME VARCHAR(200),
    ...
)
	
ex) DB2でIdentity設定 {MEMBERのMEMBER_IDにIdentity(always)} @DDL
CREATE TABLE MEMBER
(
    MEMBER_ID int generated always as identity NOT NULL PRIMARY KEY,
    MEMBER_NAME VARCHAR(200),
    ...
)

補足：DB2でInsert文の実行後に発行された連番を取得

Insert文の実行後に発行された連番を取得するSQLは以下のようになります。DBFluteを利用している場合は内部的に発行されるので意識する必要はありません。 

ex) DB2でInsert文の実行後に発行された連番を取得 @SQL
values IDENTITY_VAL_LOCAL();

ページングの実現方法

ConditionBeanではlimit処理としてfetch first n rows only構文を利用しています。offset処理はResultSetの読み飛ばしを利用します。将来的にrow_number関数を利用しての実現を検討しています(2008年1月頃)。 

ex) ConditionBeanでページング @SQL
    ... fetch first 100 rows only

OutsideSqlのManualPaging(手動ページング)を利用する場合は、row_number関数を使って以下のように実装します。 

ex) OutsideSqlのManualPaging(手動ページング)でページング @SQL
select *
  from (
  
    select ..., row_number() over(order by ...) as rn from ... where ...
	
       )
 where rn > /*$pmb.pageStartIndex*/80
   and rn <= /*$pmb.pageEndIndex*/100
 order by rn asc;

DB2でBLOG/CLOBを使う場合の注意点 @Java

DB2のJDBCドライバでは、ResultSetTypeがScrollableの場合にBLOBを取得できない(例外発生)という報告があります。 DBFluteのデフォルトはScrollableなのでDBFluteのデフォルトはForwardOnlyですが ページングを行う際はScrollableを使うことがありますので、BLOB/CLOBの含まれているテーブルは検索できない可能性があります。 もし、それでお困りの場合は以下の機能をご利用ください。

ConditionBeanやOutsideSqlのconfigure()メソッドでStatementConfigを指定します。 

ex) ConditionBeanでResultSetTypeをForwardOnlyに設定 @Java
MemberCB cb = new MemberCB();
cb.configure(new StatementConfig().typeForwardOnly());
... = memberBhv.selectList(cb);// ResultSetTypeはForwardOnlyで実行される
	
ex) OutsideSqlでResultSetTypeをForwardOnlyに設定 @Java
StatementConfig config = new StatementConfig().typeForwardOnly();
... = memberBhv.outsideSql().configure(config).selectList(...);// ResultSetTypeはForwardOnlyで実行される

実装例が、DBFluteBasicExampleにありますのでそちらぜひご覧になられて下さい。
