DBベンダごとの特徴

概要

DBFluteを利用する上でのMySQLの特徴です。

※MySQLのバージョンはMySQL-5.0以上を前提として記載しています。

JDBCタスクで利用するJDBCドライバ

JDBCドライバは「mysql-connector-java-5.0.4-bin.jar」を利用しています。

JDBCタスクの接続設定(./dfprop/databaseInfoMap.dfprop)

MySQLにつなぐJDBCタスクの接続設定は以下の通りです。 

ex) MySQLに接続{port=3306} @databaseInfoMap.dfprop
map:{
    ; driver          = com.mysql.jdbc.Driver
    ; url             = jdbc:mysql://localhost:3306/[dbname]
    ; schema          = [dbuser]
    ; user            = [dbuser]
    ; password        = [dbpassword]
}

主キーで連番の利用

MySQLは連番生成の仕組みとしてIdentityを利用します。

DBFluteでのIdentity設定

特に必要ありません。DBFluteではIdentityはJDBCタスクにおいてスキーマ情報としてIdentityの情報を取得します。特別Identityを利用するためにしなければならない設定はありません。(通常S2DaoではIDアノテーションを付ける必要がある)。

DBFluteでのInsert時のIdentity利用

Entityの主キーのプロパティに値を何も設定せずに実行すると内部的に連番が発行されます。実行後に発行された連番をそのEntityの主キーのプロパティから取得できます。 

ex) MySQLでIdentityを利用してInsert {MEMBERをInsert} @Java
Member member = new Member();
member.setMemberName("Billy Joel");
member.set... // その他主キー以外にプロパティに値を設定
memberBhv.insert(member);
Integer memberId = member.getMemberId();// 発行された連番の値を取得

補足：Create文でのIdentity設定

Create文では以下のように指定します。 

ex) MySQLでCreate文にIdentityを設定 {MEMBERのMEMBER_IDにIdentity} @DDL
CREATE TABLE MEMBER
(
	MEMBER_ID INTEGER AUTO_INCREMENT NOT NULL,
	MEMBER_NAME VARCHAR(200) NOT NULL,
	...
)

補足：Insert文の実行後に発行された連番を取得するSQL

Insert文の実行後に発行された連番を取得するSQLは以下のようになります。DBFluteを利用している場合は内部的に発行されるので意識する必要はありません。 

ex) Insert文の実行後に発行された連番を取得するSQL @SQL
select last_insert_id();

ページングの実現方法

limit句を利用しています。 

ex) ConditionBeanでページング @SQL
... limit 80, 20;

OutsideSqlのManualPaging(手動ページング)を利用する場合は以下のように実装します。 

ex) OutsideSqlのManualPaging(手動ページング)でページング @SQL
... limit /*$pmb.pageStartIndex*/80, /*$pmb.fetchSize*/20

MySQLを小文字管理にした場合の注意点

MySQLはLinux上ではではSQL文でのテーブル名やカラム名の大文字小文字を区別します。これをWindows上と同じように区別ないようにするためには、my.cnf or my.iniに以下のような設定をします。 

ex) SQL分でのテーブル名やカラム名の大文字小文字を区別しない設定 @my.cnf
[mysqld]
lower_case_table_names=1

このようにするとMySQL内部では全て小文字で管理されるようになり、SQL文での大文字で書かれたテーブル名やカラム名が実行時に小文字に変換されて処理されます。これを意識していないと、開発はローカルPCのWindowsで大文字小文字区別せずにSQLを書いていて、いざ結合テストなどでサーバのLinuxで動かしたときにSQLが動かないというような状況が発生する可能性があります。

DBFluteにおいて気をつけるべきは、上記の設定をした際に、classificationDeploymentMap.dfpropなどの設定ファイル上は小文字で記述しなければなりません。 

ex) MySQLで小文字管理にした場合のDBFluteでの区分値配置の設定 @classificationDeploymentMap.dfprop
map:{
		; $$ALL$$       = map:{suffix:_flg=Flg}
		; member        = map:{member_status_code=MemberStatus}
		; member_status = map:{member_status_code=MemberStatus}
	}

他に似たような例としてCommonColumn(共通列)の設定があります。大文字小文字を区別しないようにしている設定も存在するのですが、まだ全ての設定でやりきれていません(2008年1月頃)。

ちなみに、ConditionBeanやOutsideSql(外だしSQL)などでのプログラムからのSQLの実行に関しては、上記の設定がどうであれ特に問題はありません。

データベースエンジンの設定

DBFluteと直接関係のある話ではありませんが、MySQLのデータベースエンジンの設定の仕方を紹介します。InnoDBにするには以下のように設定します。 

ex) MySQLのデータベースエンジンをInnoDBに設定 @my.cnf
[mysqld]
default-table-type=InnoDB

トランザクション分離レベルの設定

DBFluteと直接関係のある話ではありませんが、MySQLでのトランザクション分離レベルの設定の仕方を紹介します。MySQLでは、トランザクション分離レベルはデフォルトで「Repeatable Read」になっております。他のデータベースのRepeatable Read」とは違って、ロックを取得しないため同時実行性に問題はないようですが(詳しくはMySQLのサイトで)、他のデータベースのデフォルトである「Read Committed」にしたい場合が多々あります。その場合は以下のように指定します。 

ex) MySQLのトランザクション分離レベルをRead-Committedに設定 @my.cnf
[mysqld]
transaction-isolation=Read-Committed

キャラセットの設定

DBFluteと直接関係のある話ではありませんが、MySQLでのキャラセットの設定の仕方を紹介します。 UTF-8にしたい場合は以下のように設定します。 

ex) MySQLのキャラセットを「UTF-8」に設定 @my.cnf
[mysqld]
default-character-set=utf8

SQLモードのススメ

DBFluteと直接関係のある話ではありませんが、MySQLのSQLモードを設定することをお奨めします。

例えば、MySQLは以下のような明らかに間違ったGroupByのSQLを実行してもエラーになりません。 

ex) 間違ったGroupByのSQL {SQLモードを指定することによりエラーになる} @SQL
select MEMBER_STATUS_CODE, MEMBER_NAME from MEMBER group by MEMBER_STATUS_CODE;

こういった「安全ではない割り切り」が多数存在します。SQLモードを設定することでそれらをしっかり実行時エラーにすることが可能です。SQLモードの設定は、my.cnf or my.iniに以下のように設定することが可能です。カンマ区切りで複数指定が可能です。「a, b」というように空白を一つ開けると不可なので注意して下さい(「a,b」と指定すること)。 

ex) SQLモードの設定 {SQLモードを指定することによりエラーになる} @my.cnf
[mysqld]
sql_mode="PIPES_AS_CONCAT,ANSI_QUOTES,IGNORE_SPACE,NO_DIR_IN_CREATE,STRICT_TRANS_TABLES,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION,NO_ZERO_DATE,NO_ZERO_IN_DATE,ONLY_FULL_GROUP_BY"

代表的なSQLモードを以下に紹介します。

  • PIPES_AS_CONCAT ： SQL上の文字列連結で'||'が利用可能になる
  • ONLY_FULL_GROUP_BY ： 間違ったGroupByがちゃんとエラーになる

FixedConditionの活用

MySQLはleft outer joinのon句内のサブクエリの中で結合元テーブルのカラム参照が可能なので、FixedConditionを効果的に活用できます。例えば「親 left outer join あるカラムが最大値の子」というような結合が可能になります。

例えば、会員(MEMBER)の最終ログイン日時を取得したい場合は、まずAdditionalForeignKeyで以下のように設定します。(親テーブルで最終ログイン日時というカラムを明示的に保持してないことが前提) 

ex) 「親 left outer join あるカラムが最大値の子」を設定 {会員(MEMBER)の最終ログイン日時} @additionalForeignKeyMap.dfprop
map:{
    ; FK_MEMBER_MEMBER_LOGIN_LATEST  = map:{
        ; localTableName  = MEMBER    ; foreignTableName  = MEMBER_LOGIN
        ; localColumnName = MEMBER_ID ; foreignColumnName = MEMBER_ID
        ; fixedCondition  = 

    # on句の固定条件として「結合先ログイン情報が該当会員の最新のログイン日時であること」を追加
    # 以下の変数を使うことでテーブルのAlias名がSQL実行時に自動解決。
    #   $$localAlias$$   = 基点テーブルのAlias名
    #   $$foreignAlias$$ = 結合テーブルのAlias名
    $$foreignAlias$$.LOGIN_DATETIME = (select max(LOGIN_DATETIME) from MEMBER_LOGIN where MEMBER_ID = $$localAlias$$.MEMBER_ID)

        ; fixedSuffix = AsLatest
    }
}

すると、上記設定後に自動生成し直すと、以下のようにConditionBeanで最終ログイン日時を取得することができます。 

ex) 「親 left outer join ある値が最新の子」を設定 {会員(MEMBER)の最終ログイン日時} @Java
MemberCB cb = new MemberCB();
cb.setupSelect_MemberLoginAsLatest();
List<Member> memberList = memberBhv.selectList(cb);
for (Member member : memberList) {
    MemberLogin memberLogin = member.getMemberLoginAsLatest();
	... = memberLogin.getLoginDatetime();// 最終ログイン日時を取得
}
