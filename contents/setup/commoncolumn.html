共通カラム(CommonColumn)

DBFluteには以下のような共通カラム(CommonColumn)を認識してInsertやUpdate時に自動登録する機能があります。

例えば以下のようなカラムです。

  • 登録日時 : REGISTER_DATETIME
  • 登録ユーザ : REGISTER_USER
  • 登録プロセス : REGISTER_PROCESS
  • 更新日時 : UPDATE_DATETIME
  • 更新ユーザ : UPDATE_USER
  • 更新プロセス : UPDATE_PROCESS

*Seasarカンファレンス「現場ソリューション DBFlute」においても紹介されました。

運用者のための共通カラム

構成はアプリケーションによってまちまちだと思われます。どれだけ運用時にデータのトレーサビリティが必要になるか次第ですが、運用者にとってこのカラムはトラブル時にとてもとても大事な情報となります。

通常、日時とユーザだけのパターンが多いですが、プロセス(画面やバッチ処理など)も一つのテーブルを複数のプロセスがアクセスする場合はとても重要です。「このレコードいつ誰が更新したのかはわかるけど、どの画面から更新したのかがわからない」って状況も生まれかねません。どういった情報を残しておくべきかをアプリケーションの特性を考慮して適切にDB設計することをお奨めします。

開発者からみた共通カラム

しかし、開発者にとっては、常に全てのテーブルのInsertやUpdate時においてこれらカラムを指定しなければなりません。それが「煩雑」であるというだけで済むのであれば開発者が頑張れば良いだけなのですが、何度も同じことを繰り返していくうちに指定の仕方を間違えたり、指定を忘れて更新がされなかったりと後で運用者を泣かせる結果になりかねません。この非機能要件的なカラムは結合テストや運用テストでも厳密なテストがされない場合もあります(本来良くないですが現実問題として)。なので、開発者のミスがそのまま運用に直結しやすい部分でもあるのです。

DBFluteでの解決

これらのカラムは規則的に値が定まります。よって、DBFluteにその規則性を設定することで自動でInsertやUpdate時に自動で設定されるようにします。 

ex) 会員名称を「Billy Joel」に更新と同時に共通カラムも更新 @Java
Member member = new Member();
member.setMemberName("Billy Joel");

// これらは不要
// member.setUpdateDatetime(...);
// member.setUpdateUser(...);

member.update(member); // 共通カラムは内部で自動設定される

日時に関してはデータベースの機能であるトリガで対処も可能ですが、ユーザやプロセスは基本的にアプリケーションのセッション情報から取得しなければならないため、根本的な解決にはなりません。

共通カラムの設定手順

1. 外だしビルドプロパティに共通カラムを設定

DBFluteクライアント配下の「./dfprop/commonColumnMap.dfprop」に以下のような設定をして再自動生成をして下さい。(*dfpropファイルはビルドプロパティの外だし形式の設定のことを言います) 

ex) 共通カラムの規則性を設定(Javaの場合) @commonColumnMap.dfprop
map:{
    ; commonColumnMap = map:{
        ; REGISTER_DATETIME=TIMESTAMP ; REGISTER_USER=VARCHAR ; REGISTER_PROCESS=VARCHAR
        ; UPDATE_DATETIME=TIMESTAMP   ; UPDATE_USER=VARCHAR   ; UPDATE_PROCESS=VARCHAR
    }
    ; beforeInsertMap = map:{
        ; REGISTER_DATETIME = $$AccessContext$$.getAccessTimestampOnThread()
        ; REGISTER_USER     = $$AccessContext$$.getAccessUserOnThread()
        ; REGISTER_PROCESS  = $$AccessContext$$.getAccessProcessOnThread()
        ; UPDATE_DATETIME   = entity.getRegisterDatetime()
        ; UPDATE_USER       = entity.getRegisterUser()
        ; UPDATE_PROCESS    = entity.getRegisterProcess()
    }
    ; beforeUpdateMap = map:{
        ; UPDATE_DATETIME   = $$AccessContext$$.getAccessTimestampOnThread()
        ; UPDATE_USER       = $$AccessContext$$.getAccessUserOnThread()
        ; UPDATE_PROCESS    = $$AccessContext$$.getAccessProcessOnThread()
    }
}

C#の場合は、以下の点が異なります。

  • 「$$AccessContext$$.getAccess...」→「$$AccessContext$$.GetAccess...」
  • 「entity.getRegisterDatetime()」→「entity.RegisterDatetime」

「commonColumnMap」にて、そもそも共通カラムはどのカラムなのかを指定します。 そして、それぞれのカラムにInsert/Update時にどんな値を格納するかを「beforeInsertMap」/「beforeUpdateMap」にて設定します。

「$$AccessContext$$」は、DBFluteが生成するallcommonパッケージのAccessContextのThreadLocalを保持するクラスです。このクラスから共通カラムに格納する値を取得します。

commonColumnMapのカラムタイプ

commonColumnMapのカラムタイプは、以下の「DBFlute Type」からそのカラムが当てはまるタイプを選んで下さい。

2. AccessContext(ThreadLocal)に値を格納

それぞれの共通カラムがAccessContextから値を取得するようになりましたので、後はAccessContextに値を格納すればおしまいです。

格納タイミングとしてアプリケーションの構成次第ですが、画面アプリであればPageクラスのInterceptor、バッチアプリであれば起動時の処理(仕掛けるタイミングがあればInterceptorでOK)などになります。 

ex) InterceptorでAccessContextに値を格納 @Java
public Object invoke(MethodInvocation invocation) throws Throwable {
    try {
        // もしアプリケーションで、日時を取得する統一したインターフェースがあればそちらから取得すること。
        String accessTimestamp = new Timestamp(System.currentTimeMillis());
        
        // セッション上のログインユーザから取得すること。
        // ログインしていない場合のことも考慮すること。
        String accessUser = session.getLoginUser();
        
        // ここではPageクラスの名前をそのまま利用している。
        // アプリケーションでプロセスを判別できる適切な名前を設定すること。
        // DBのカラムサイズに注意(サイズオーバーしないよに)
        String accessProcess = getClass().getName();
        
        AccessContext context = new AccessContext();
        context.setAccessTimestamp(accessTimestamp);
        context.setAccessUser(accessUser);
        context.setAccessProcess(accessProcess);
        AccessContext.setAccessContextOnThread(context);
        
        return invocation.proceed();
    } finally {
        // 最後はしっかりクリアすること。
        AccessContext.clearAccessContextOnThread();
    }
}

このクラスはアプリケーションで独自に作成して、適切なタイミングで実行して下さい。

3. プログラムから実行して確認

実際に実行してみて確認してみて下さい。以下のような自動設定前と後を比較するログが出るはずです。 

ex) 共通カラムが自動設定されたときのログ @Log
...Filtering entity of INSERT about the column columns of MEMBER: entity=map:{MEMBER_ID=}

補足：論理削除フラグの設定

よくあるパターンとして、論理削除フラグが共通カラムの仲間入りをした場合のことを考えてみます。

論理削除フラグは、Insert時には必ずfalse(論理削除されていない有効な状態)で登録することが多いです。また、論理削除しない更新時は値はそのままで、論理削除する更新時には明示的に指定します。よって、共通カラムの設定としては以下のようになります。 

ex) 共通カラムに論理削除フラグを仲間入り(フラグの表現が'0 or 1'の場合) @commonColumnMap.dfprop
map:{
    ; commonColumnMap = map:{
        ; REGISTER_DATETIME=TIMESTAMP ; REGISTER_USER=VARCHAR ; REGISTER_PROCESS=VARCHAR
        ; UPDATE_DATETIME=TIMESTAMP   ; UPDATE_USER=VARCHAR   ; UPDATE_PROCESS=VARCHAR
        ; LOGICAL_DELETE_FLG=INTEGER
    }
    ; beforeInsertMap = map:{
        ; REGISTER_DATETIME = $$AccessContext$$.getAccessTimestampOnThread()
        ; REGISTER_USER     = $$AccessContext$$.getAccessUserOnThread()
        ; REGISTER_PROCESS  = $$AccessContext$$.getAccessProcessOnThread()
        ; UPDATE_DATETIME   = entity.getRegisterDatetime()
        ; UPDATE_USER       = entity.getRegisterUser()
        ; UPDATE_PROCESS    = entity.getRegisterProcess()
        ; LOGICAL_DELETE_FLG = 0
    }
    ; beforeUpdateMap = map:{
        ; UPDATE_DATETIME   = $$AccessContext$$.getAccessTimestampOnThread()
        ; UPDATE_USER       = $$AccessContext$$.getAccessUserOnThread()
        ; UPDATE_PROCESS    = $$AccessContext$$.getAccessProcessOnThread()
    }
}

ポイントは、Insert時に固定でfalseを格納しているところと、Update時には何を設定していないことです。これによりプログラム上では本当に論理削除する更新時のときのみ論理削除フラグを意識すればよいことになります。

補足：区分値機能との併用

先ほどの論理削除フラグの例で、そのカラムに区分値が関連づけられている場合は上記のように「0」という値をハードコードするのはよくありません。以下のように記述が可能です。 

ex) 共通カラムに論理削除フラグを仲間入り(フラグの表現が区分値で解決) @commonColumnMap.dfprop
map:{
    ; commonColumnMap = map:{
        (略)
    }
    ; beforeInsertMap = map:{
        (略)
        ; LOGICAL_DELETE_FLG = $entity.classifyLogicalDeleteFlgFalse()$$Semicolon$$
    }
    ; beforeUpdateMap = map:{
        (略)
    }
}
  • 「$」を付けることで、「代入」ではなく「単なる指定されたメソッドの実行」になります。
  • 「$$Semicolon$$」はセミコロンとなります。(設定ファイルのデリミタが「;」なので変数にしています)
